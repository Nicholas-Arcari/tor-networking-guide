# Lab 04 - Creating and Testing a v3 Onion Service

A hands-on exercise to configure a v3 Onion Service, verify its
operation, implement client authentication, and understand the
cryptographic key structure.

**Estimated time**: 35-45 minutes
**Prerequisites**: Lab 01 completed, root access, web server (nginx or python3)
**Difficulty**: Intermediate-Advanced

## Objectives

By the end of this lab, you will have:
1. A working v3 Onion Service with a `.onion` address
2. Understood the structure of the ed25519 keys generated by Tor
3. Configured client authentication (x25519)
4. Applied basic hardening to the hidden service
5. Tested end-to-end connectivity via Tor Browser and curl

## Phase 1: Local web server

```bash
# Option A: Python (simplest for the lab)
mkdir -p /var/www/onion-lab
echo "<h1>Onion Service Lab</h1><p>Funziona! Sei connesso via Tor.</p>" \
    > /var/www/onion-lab/index.html

# Start the server on port 8080 (localhost only)
cd /var/www/onion-lab
python3 -m http.server 8080 --bind 127.0.0.1 &
HTTP_PID=$!

# Verify that the server responds
curl -s http://127.0.0.1:8080/
# Expected output: <h1>Onion Service Lab</h1>...
```

```bash
# Option B: nginx (more realistic)
sudo apt install -y nginx
sudo tee /etc/nginx/sites-available/onion-lab << 'EOF'
server {
    listen 127.0.0.1:8080;
    server_name localhost;
    root /var/www/onion-lab;
    index index.html;

    # Security headers
    add_header X-Content-Type-Options nosniff;
    add_header X-Frame-Options DENY;
    add_header Referrer-Policy no-referrer;

    # Disable logging for the hidden service
    access_log off;
    error_log /dev/null;
}
EOF

sudo ln -sf /etc/nginx/sites-available/onion-lab /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx

curl -s http://127.0.0.1:8080/
```

**Verification**: `curl http://127.0.0.1:8080/` returns the HTML page.

> **Security note**: the server listens only on `127.0.0.1` - it is not
> reachable from the outside. Only Tor can reach it via the Onion Service.

## Phase 2: Onion Service configuration

```bash
# Add the configuration to torrc
sudo tee -a /etc/tor/torrc << 'EOF'

# === Lab 04 - Onion Service ===
HiddenServiceDir /var/lib/tor/onion-lab/
HiddenServicePort 80 127.0.0.1:8080
HiddenServiceVersion 3
EOF

# Restart Tor
sudo systemctl restart tor@default.service

# Wait for Tor to generate the keys and publish the descriptor
sleep 5

# Read the generated .onion address
sudo cat /var/lib/tor/onion-lab/hostname
# Output: a 56-character address + .onion
# Example: dz4k6x2xbv5aomht...d.onion
```

```bash
# Save the address in a variable for subsequent tests
ONION_ADDR=$(sudo cat /var/lib/tor/onion-lab/hostname)
echo "Your Onion Service: $ONION_ADDR"

# Verify that Tor is active with the service
sudo journalctl -u tor@default.service --no-pager | tail -5
# You should see: "Registered new hidden service"
```

**Verification**: a `hostname` file exists in `/var/lib/tor/onion-lab/` with a
56-character v3 `.onion` address.

## Phase 3: Verification and connection

```bash
# Test 1: connection via curl and SOCKS5
ONION_ADDR=$(sudo cat /var/lib/tor/onion-lab/hostname)
curl --socks5-hostname 127.0.0.1:9050 -s --max-time 60 "http://$ONION_ADDR"
# Expected output: <h1>Onion Service Lab</h1>...
# NOTE: the first connection may take 15-30 seconds

# Test 2: connection via torsocks
torsocks curl -s --max-time 60 "http://$ONION_ADDR"
# Same expected output

# Test 3: verification with proxychains
proxychains curl -s --max-time 60 "http://$ONION_ADDR"
```

```bash
# Test 4: verification in Tor Browser
# Open Tor Browser and visit the .onion address
echo "Open Tor Browser and visit: http://$ONION_ADDR"

# Test 5: timing - measure the latency
time curl --socks5-hostname 127.0.0.1:9050 -s --max-time 60 \
    -o /dev/null -w "HTTP %{http_code} in %{time_total}s\n" "http://$ONION_ADDR"
# Typical output: HTTP 200 in 2.5-5.0s (6 hops: 3 client + 3 service)
```

**Verification**: the page is reachable from all three methods. The latency
is higher than normal (6 hops vs 3).

## Phase 4: Key structure

```bash
# Examine the files generated by Tor
sudo ls -la /var/lib/tor/onion-lab/
# Output:
# hostname                - public .onion address
# hs_ed25519_public_key   - ed25519 public key (32 bytes)
# hs_ed25519_secret_key   - ed25519 private key (64 bytes)

# Analyze the keys
sudo xxd /var/lib/tor/onion-lab/hs_ed25519_public_key | head -5
# The first 32 bytes after the header "== ed25519v1-public: type0 ==\x00\x00\x00"
# are the actual public key

sudo xxd /var/lib/tor/onion-lab/hs_ed25519_secret_key | head -5
# Header: "== ed25519v1-secret: type0 ==\x00\x00\x00"
# Followed by 64 bytes of the private key

# Decode the .onion address
ONION_ADDR=$(sudo cat /var/lib/tor/onion-lab/hostname)
echo "Address: $ONION_ADDR"
echo "Length (without .onion): ${#ONION_ADDR%%.*}"
# 56 characters = base32 of (32-byte pubkey + 2-byte checksum + 1-byte version)
```

```bash
# Verify the relationship between the public key and the address
# The v3 .onion address is: base32(pubkey || checksum || version)
python3 << 'PYEOF'
import base64, hashlib

# Read the public key (skip 32-byte header)
with open("/var/lib/tor/onion-lab/hs_ed25519_public_key", "rb") as f:
    data = f.read()
    pubkey = data[32:]  # 32 byte dopo l'header

# Calculate v3 address
version = bytes([3])
checksum = hashlib.sha3_256(
    b".onion checksum" + pubkey + version
).digest()[:2]

onion_bytes = pubkey + checksum + version
address = base64.b32encode(onion_bytes).decode().lower() + ".onion"
print(f"Calcolato:  {address}")

# Compare with the one generated by Tor
with open("/var/lib/tor/onion-lab/hostname") as f:
    hostname = f.read().strip()
print(f"Da Tor:     {hostname}")
print(f"Match: {address == hostname}")
PYEOF
```

**Exercise**: why is the private key the most critical secret? What
happens if someone obtains it?

## Phase 5: Client authentication

```bash
# Generate an x25519 key pair for client authentication
# Requires openssl >= 1.1 or the tor tool
# Method with Python (works everywhere):

python3 << 'PYEOF'
import os, base64

# Generate x25519 private key (32 random bytes, clamped)
privkey = bytearray(os.urandom(32))
privkey[0]  &= 248
privkey[31] &= 127
privkey[31] |= 64

# For the public key, we need the x25519 curve
# We use the cryptography library if available
try:
    from cryptography.hazmat.primitives.asymmetric.x25519 import X25519PrivateKey
    from cryptography.hazmat.primitives import serialization

    key = X25519PrivateKey.from_private_bytes(bytes(privkey))
    pubkey = key.public_key().public_bytes(
        serialization.Encoding.Raw,
        serialization.PublicFormat.Raw
    )
except ImportError:
    print("ERRORE: installa python3-cryptography")
    print("  sudo apt install python3-cryptography")
    exit(1)

priv_b32 = base64.b32encode(privkey).decode().rstrip("=")
pub_b32 = base64.b32encode(pubkey).decode().rstrip("=")

print(f"# Chiave PRIVATA (per il client) - file: client.auth_private")
print(f"# <onion-addr-no-.onion>:descriptor:x25519:<base32-privkey>")
print(f"PRIVKEY: {priv_b32}")
print()
print(f"# Chiave PUBBLICA (per il server) - file: <nome>.auth")
print(f"# descriptor:x25519:<base32-pubkey>")
print(f"PUBKEY: {pub_b32}")

# Save the files
with open("/tmp/lab04-privkey.txt", "w") as f:
    f.write(priv_b32)
with open("/tmp/lab04-pubkey.txt", "w") as f:
    f.write(pub_b32)
PYEOF
```

```bash
# Configure the SERVER SIDE (client authorization)
PUBKEY=$(cat /tmp/lab04-pubkey.txt)
sudo mkdir -p /var/lib/tor/onion-lab/authorized_clients
echo "descriptor:x25519:$PUBKEY" | \
    sudo tee /var/lib/tor/onion-lab/authorized_clients/lab-client.auth

# Restart Tor to activate authentication
sudo systemctl restart tor@default.service
sleep 5

# Now WITHOUT authentication, the service is NOT reachable
ONION_ADDR=$(sudo cat /var/lib/tor/onion-lab/hostname)
curl --socks5-hostname 127.0.0.1:9050 -s --max-time 30 "http://$ONION_ADDR"
# Expected output: error or empty page (descriptor cannot be decrypted)
```

```bash
# Configure the CLIENT SIDE
ONION_ADDR=$(sudo cat /var/lib/tor/onion-lab/hostname)
ONION_NAME="${ONION_ADDR%.onion}"
PRIVKEY=$(cat /tmp/lab04-privkey.txt)

# Create the client authentication file
sudo mkdir -p /var/lib/tor/onion_auth
echo "${ONION_NAME}:descriptor:x25519:${PRIVKEY}" | \
    sudo tee /var/lib/tor/onion_auth/lab-service.auth_private

# Add the directive to torrc (if not already present)
grep -q "ClientOnionAuthDir" /etc/tor/torrc || \
    echo "ClientOnionAuthDir /var/lib/tor/onion_auth" | \
    sudo tee -a /etc/tor/torrc

# Restart Tor
sudo systemctl restart tor@default.service
sleep 5

# Now WITH authentication, the service is reachable
curl --socks5-hostname 127.0.0.1:9050 -s --max-time 60 "http://$ONION_ADDR"
# Expected output: <h1>Onion Service Lab</h1>...
```

**Exercise**: what happens if you remove the `.auth` file from the server but
the client still has the `.auth_private` file? And vice versa?

## Phase 6: Vanity address (optional)

```bash
# mkp224o generates .onion addresses with a chosen prefix
# WARNING: the longer the prefix, the more time it takes

sudo apt install -y gcc libsodium-dev make autoconf git
git clone https://github.com/cathugger/mkp224o.git /tmp/mkp224o
cd /tmp/mkp224o
./autogen.sh && ./configure && make

# Generate an address starting with "test"
# (4 characters = a few seconds, 6+ = hours/days)
./mkp224o -d /tmp/vanity-keys test -n 1 -S 10
ls /tmp/vanity-keys/

# Inside the folder you will find: hostname, hs_ed25519_public_key, hs_ed25519_secret_key
# You can copy them in place of the ones generated by Tor (back up first!)
```

**Note**: vanity addresses are cosmetic. They do not add security
and the generation time grows exponentially with the prefix length.

## Phase 7: Service hardening

```bash
# 1. Verify key permissions
sudo ls -la /var/lib/tor/onion-lab/
# hs_ed25519_secret_key should be: -rw------- (600) owner: debian-tor
sudo stat -c "%a %U:%G" /var/lib/tor/onion-lab/hs_ed25519_secret_key
# Expected output: 600 debian-tor:debian-tor

# 2. Restrict the web server to localhost only
ss -tlnp | grep 8080
# Output: should show 127.0.0.1:8080 (not 0.0.0.0:8080)

# 3. Add security headers (if using nginx)
# → Already configured in Phase 1 (Option B)

# 4. Limit connections with HiddenServiceMaxStreams
# Add to torrc in the onion service section:
#   HiddenServiceMaxStreams 20
#   HiddenServiceMaxStreamsCloseCircuit 1

# 5. Isolate the service with systemd sandboxing
sudo systemctl edit tor@default.service << 'EOF'
[Service]
NoNewPrivileges=yes
ProtectHome=yes
ProtectSystem=strict
ReadWritePaths=/var/lib/tor /var/log/tor
EOF
sudo systemctl daemon-reload && sudo systemctl restart tor@default.service
```

```bash
# 6. Verify that the service does not expose information
curl --socks5-hostname 127.0.0.1:9050 -s -I "http://$ONION_ADDR" | head -10
# Check that there is NO:
# - Server: nginx/x.x.x  (use "server_tokens off;" in nginx)
# - X-Powered-By: ...
# - Software version information
```

## Troubleshooting

### The .onion address is not generated

```bash
# Check the Tor logs
sudo journalctl -u tor@default.service --no-pager | grep -i "hidden\|error"

# Common cause: incorrect permissions on the directory
sudo ls -la /var/lib/tor/onion-lab/
# The directory should be: owner debian-tor, permissions 700
sudo chown -R debian-tor:debian-tor /var/lib/tor/onion-lab/
sudo chmod 700 /var/lib/tor/onion-lab/

# Common cause: syntax error in torrc
sudo tor --verify-config
# If it shows errors, fix the torrc and restart
```

### curl to .onion times out

```bash
# The first connection may take 30-60 seconds (6 hops to build)
# Increase the timeout:
curl --socks5-hostname 127.0.0.1:9050 -s --max-time 120 "http://$ONION_ADDR"

# If it persists, verify that the local web server is running:
curl -s http://127.0.0.1:8080/
# If this fails → the server is not running, it is not a Tor problem

# Verify that Tor has published the descriptor:
sudo journalctl -u tor@default.service | grep "Registered new hidden service"
# If it does not appear → the torrc has not been reloaded
sudo systemctl restart tor@default.service
```

### Client authentication does not work

```bash
# Error: the service becomes unreachable after adding auth

# 1. Verify .auth file format (server side)
sudo cat /var/lib/tor/onion-lab/authorized_clients/lab-client.auth
# Correct format: descriptor:x25519:<base32_pubkey>
# Common error: extra spaces or newlines in the file

# 2. Verify .auth_private file format (client side)
sudo cat /var/lib/tor/onion_auth/lab-service.auth_private
# Correct format: <onion_without_.onion>:descriptor:x25519:<base32_privkey>

# 3. Restart Tor after every change to auth files
sudo systemctl restart tor@default.service
sleep 10  # wait for descriptor republication
```
